# ISA — template_pitch_deck

## Problem

The repo's public exemplar roster (`projects/templates/`) has no template for pitch-deck generation. Every existing exemplar renders to manuscript-style PDF via the LaTeX/pandoc pipeline (`infrastructure/rendering/`) or, in two cases (`template_newspaper`, `template_storybook`), a bespoke ReportLab layout — but none produces a slide deck, none emits PPTX, and there is no infrastructure module for slide-shaped content at all. Meta-science and science-integrity work (the Active Inference Institute, COGSEC, this very template project) regularly needs investor/stakeholder/partner pitch decks, and today that means hand-building slides outside the repo's reproducibility guarantees — no token validation, no deterministic regeneration, no coverage gate, no public exemplar to fork from.

## Vision

A user runs one command and gets a fully reproducible `output/template_pitch_deck/` folder containing six real artifacts — short/medium/long × PDF/PPTX — every one of them built from the same token-resolved content source, none of them containing a stray `{{TOKEN}}` literal, none of them reading as generic pitch-deck cliché ("synergy," "disruptive," "unicorn," stock rocket-ship clip art prose). The flagship example deck pitches `template_template` itself, honestly, as meta-science research infrastructure for a science-integrity group — the pitch a real Active Inference Institute / COGSEC-style org could actually hand to a funder. Opening the long-form PDF and the PPTX side by side, a reviewer should not be able to tell which renderer birthed which — same numbers, same narrative arc, same slide count proportions, both professionally laid out, neither embarrassing.

## Out of Scope

- **No live presentation engine** (no reveal.js/impress.js HTML slideshow, no in-browser presenter mode) — this template only emits static PDF and PPTX files. A future template may add HTML slides; not this one.
- **No AI-generated slide imagery/photos** — the deck uses typographic layout, simple vector shapes (rects, lines, rules) via ReportLab/python-pptx primitives, and repo-generated matplotlib figures where a chart genuinely helps (e.g. a roadmap timeline) — never stock photography or externally-fetched images (keeps the pipeline offline-reproducible and license-clean).
- **Not a general slide-authoring DSL** — the template ships with one deterministic content schema (`manuscript/deck_content.yaml` + `manuscript/config.yaml`) sufficient to author the `template_template` pitch and any future pitch of similar shape; it is not attempting to become a full presentation-authoring language (no per-slide arbitrary Python callbacks, no plugin system).
- **No animation/transition metadata** in the PPTX (no fade/wipe build effects) — PPTX output is static-slide, matching the PDF 1:1.
- **No LLM-authored deck content in this template's core pipeline** — the flagship `template_template` pitch content is hand-authored (token-templated) prose checked into the repo, not generated at render time by an LLM call; this keeps the exemplar deterministic and Ollama-optional, consistent with every other public exemplar's core-pipeline guarantee.
- **Not retrofitting PPTX into the standard manuscript pipeline** for the other 17 templates — this is a new, additive `infrastructure/rendering/pptx_deck.py`-style module and a new `infrastructure/rendering/slide_deck.py` PDF path used only by `template_pitch_deck` (and available to any future deck-shaped project); it does not change how `template_code_project`-style manuscripts render.

## Principles

- **Thin orchestrator, as everywhere else in this repo**: all deck-layout, token-resolution, and validation logic lives in `infrastructure/rendering/` (and `infrastructure/documentation/` for token substitution if reused); `projects/templates/template_pitch_deck/scripts/*.py` only calls into it and handles I/O.
- **One content source, N render targets**: the same resolved `DeckContent` object feeds both the PDF renderer and the PPTX renderer, and feeds all three length variants via a `SlideBudget` filter — never three independently hand-maintained content files.
- **Honesty over ceremony**: "validated" means an executable auditor (`infrastructure.validation` addition or a project-local `src/token_audit.py` built on infra primitives) that fails loudly if any `{{TOKEN}}` literal survives into a rendered artifact — not a comment claiming it was checked (memory: `gotcha-python-replace-silent-nomatch-unverified`, `gotcha-manuscript-honesty-tokens-need-real-injection-call`).
- **Non-cliché is a testable constraint, not a vibe**: a denylist-style lint (extendable, e.g. "synergy", "disruptive", "unicorn", "rocket ship", "10x", "game-changing", "revolutionize") runs over resolved slide text before render and fails the build on a hit — content quality gets the same enforcement as coverage.
- **No mocks**: deck-content tests build real `DeckContent`/`Slide` objects and assert on real rendered file bytes/structure (PDF page count via a real PDF library read, PPTX slide count via `python-pptx` read-back) — never `unittest.mock`.
- **Reproducibility**: fixed content + fixed config ⇒ byte-stable slide counts and deterministic filenames on every run; no wall-clock-dependent content (timestamps in the metadata only, following `STEGANOGRAPHY_DETERMINISTIC`-style patterns already in the repo).

## Constraints

- Must sit under `projects/templates/template_pitch_deck/` with the standard `src/`, `tests/`, `scripts/`, `manuscript/`, `output/`, `pyproject.toml` layout (per root `CLAUDE.md` "Standard Project Layout"), so it renders through the existing multi-project discovery (`infrastructure/project/discovery.py`) without special-casing.
- New rendering capability (`infrastructure/rendering/slide_deck.py`, `infrastructure/rendering/pptx_deck.py`) must be added under `infrastructure/rendering/` — the project's own `src/` may hold only pitch-deck-domain content models (`Slide`, `DeckContent`, `Section`), never layout/PDF/PPTX drawing code (thin-orchestrator invariant, root `CLAUDE.md`).
- PPTX generation requires a new dependency (`python-pptx`); it must be added as an optional dependency group (e.g. `rendering-pptx` or folded into the existing `rendering` group in root `pyproject.toml`) exactly like `discopy`/`steganography` are opt-in groups today — never a hard dependency of the default `uv sync`.
- Must pass the confidentiality/registration gates that already exist for every public exemplar: `scripts/audit/check_template_drift.py`, `scripts/audit/check_tracked_all.py` (via `git_guards.py`), `scripts/audit/lint_docs.py`, and must appear in `docs/_generated/active_projects.md` (generated by `infrastructure.project.public_scope`), the root `README.md` template roster, and the root `CLAUDE.md` "Public active projects" bullet list (this file — already partially updated by the user's own recent edit; must add the new bullet).
- Per-project coverage floor is 90% (project `src/`) with no-mocks testing (`pytest-httpserver` / real file fixtures only), and any new `infrastructure/rendering/*.py` module needs the infra 60% floor honored in the combined `tests/infra_tests/` suite.
- Must carry a `.agents/skills/template_pitch_deck/SKILL.md` matching the format already shipped by every other template (per root `CLAUDE.md` "Discoverable per-template skills").
- Must carry the `PUBLISHING-STATUS` CI-drift-gated block used by the other 13 rolled-out templates (memory: `project-template-publishing-rollout-2026-06-30`).
- Flagship content must be about `template_template` (`projects/templates/template_template/`) specifically — its actual README/AGENTS.md/DESIGN.md describe an "autopoietic meta-template that introspects infrastructure and public exemplar roster" — the pitch must be factually grounded in what that project actually does, not invented capabilities (no fabricated metrics/customers/revenue — this is a research-infrastructure pitch, not a startup pitch, so token-resolved facts must trace to the real repo: project counts, coverage numbers, CI matrix, DOI/archival story where real).
